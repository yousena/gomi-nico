## 0. デザイン原則（5カ条）

1. **親しみ×信頼**: 行政情報の正確さと、生活道具としての親しみやすさを両立する。役所っぽくしない、ふざけすぎない
2. **3秒でわかる**: 開いた瞬間に「今日のごみ」「これは何ごみ」が伝わる。読ませない、見せる
3. **引き算**: 装飾より余白とジャンプ率で構造を作る。1画面1メッセージ
4. **色には意味がある**: カテゴリ色は分別の意味を背負う。装飾目的でカテゴリ色を流用しない
5. **全員が使える**: シニアも子どももストレスなく。WCAG 2.1 AA / JIS X 8341-3 を下回らない

**禁則**: カテゴリ色を装飾に流用しない（例: 見出し飾りに可燃の赤を使う等）。逆に、カテゴリを表す場所で任意の色を使わない。

**アイコンも自治体非依存の共通資産（v1.7・2026-07-21合意）**: `TYPE_STYLE`の`img`（`/icons/*.svg`）はLP（`index.html`）・アプリ本体（`{city}/index.html`）・今後追加する自治体すべてで同じファイルを共有し、自治体ごとに差し替えない。

- **経緯**: 「志木市のSVGアイコンとLPは同じものだが、今後は分けるべきか」「他自治体を追加する際も共通か自治体ごとか」という相談を受け、既存実装を確認したところLPとshiki側は既に`/icons/moeru.svg`等を物理的に同一ファイルとして共有していた。この状態を追認・明文化する形で方針を確定した
- **理由**:
  1. 上記のカテゴリ色ルール（自治体ごとに割り当てを変えない）と同じ理屈が当てはまる。燃える・燃えない・資源・危険等のカテゴリは自治体固有ではなく普遍的な概念であり、色を固定するならアイコンだけ変える理由がない
  2. LPからアプリ本体、ある自治体から別の自治体への遷移で同じアイコンが出ることで、「同じ仕組みの続きである」という一貫性をユーザーに伝えられる（アイコンがバラバラだと超・直感型UXの方針と矛盾する）
  3. 新自治体追加のたびにアイコンを作り直す必要がなく、運用コスト・品質のブレを抑えられる（§5-5「新自治体追加時は`brand_color`のみで差別化」と整合）
- **例外**: フェーズ2の自治体との有償契約で自治体側ブランドのアイコンを求められた場合は、`TYPE_STYLE`本体を汚さない白ラベル的な差し替えオプションとして個別対応を検討する（デフォルト運用は常に共通アイコン）
- **新カテゴリが必要な場合**: 既存の`moeru`/`moenai`/`recycle`等で表現できない分別区分が出てきたときは、自治体ごとの亜種を作るのではなく`TYPE_STYLE`に新しいキー（例: 既存の`cardboard`・`kami_pack`のように）を追加し、全自治体で使える共通カテゴリとして管理する

### 1-4. スペーシング・形状

- **余白は8pxの倍数**（例外は4px刻みの微調整のみ）
- セクション: `padding: 64px 24px`（`.sec`）／注意帯など: `56px 24px`（`.sec-tight`）
- セクション見出しブロックの下余白: 32〜36px ／ カード間gap: 12〜16px
- **角丸**: pill `999px`（バッジ・タグ・**CTAボタン**）／ カード大 `20px` ／ 統計カード `16px` ／ アイコン `14px` ／ クイズ選択肢など小要素 `12px`
- **波形ディバイダー**: コントラストの強いセクション境界（ダーク帯・注意帯の前後）には高さ32〜36pxの波形SVGを入れる（v1.3で24〜26pxから深化。振幅が浅いと境界がぼやけて見えるため）。上のセクション色を地に、下のセクション色で波を塗る。連続する波は位相を反転させて単調さを避ける。すべての境界に入れない（多用すると騒がしくなる）
- **影の2階層**: 標準カード（FAQ・もっと知りたい・特徴グリッド・数字で見るごみの統計カード等）は `shadow-sm`／背景オーバーレイのみで、強い影は付けない。**強調シャドウ`--gn-shadow-elevated`（`0 8px 20px rgba(28,28,30,0.12)`）は「実際にタップ操作を伴う要素」だけに限定**する（例: クイズカード、CTA）。影＝タップできる、という誤ったアフォーダンスを避けるため、純粋な情報表示カードには使わない（v1.4で方針修正）。CTAは従来通り `0 4px 14px rgba(0,168,107,0.3)`
- **あしらいキャラ（試験導入）**: ヒーローで確立した「顔つきごみ袋」の意匠を、ページ内1〜2箇所に限定して再利用してよい（例: クイズ見出し脇・締めメッセージ）。密度を上げすぎない。効果検証後、v1.4以降で本格導入の可否を判断する

## 2. コンポーネント

### 2-1. セクション見出し（必須の型）
```html
<p class="eyebrow">ENGLISH LABEL</p>
<h2 class="sec-title">日本語の見出し。</h2>
<p class="sec-lead">1〜2行のリード文（任意）。</p>
```
中央揃え。eyebrowはブランド緑（ダーク帯上では `#8FD9B6`）。

### 2-2. ボタン
| 種別 | 仕様 |
|---|---|
| Primary | ベタ緑 `--gn-brand`・白文字・高さ52px・角丸14px・CTA影。**1画面に1〜2個まで** |
| Secondary | 白地+緑枠1.5px・緑文字・高さ44〜52px |
| テキストリンク | 緑・下線 or 矢印つき。タップ領域は44px確保（padding/min-heightで） |
| 状態 | Default / Active(scale .97) / Disabled(不透明度・カーソル)を最低限定義 |

### 2-3. カード
- 白背景 + `shadow-sm`（枠線は原則不要。使う場合 `--gn-line`）
- 白背景セクション内では `--gn-bg-card-tint` の淡色カード
- 構造: アイコン(左) + 見出し + 説明 の横型、または中央揃え縦型（グリッド用）
- **強調シャドウ**（`--gn-shadow-elevated`）はタップ操作を伴うカードのみ（1-4節参照）。純粋な表示用カードには使わない

### 2-4. ごみ種別アイコン（3系統の役割分担・厳守）
| 系統 | 用途 | 仕様 |
|---|---|---|
| 自作SVG（/icons/） | ごみカテゴリの表現 | LP: 60×60px角丸14px背景なし ／ アプリ: コンテナ44〜56px+淡背景（詳細は下表） |
| 自作SVGイラスト | 世界観・説明図解（ヒーロー、3ポイント等） | インラインSVG。フラット・表情OK・ブランドパレット準拠 |
| Material Symbols | 機能アイコン（検索・矢印・カレンダー等） | `.ms`(塗り)/`.ms-o`(線)。ごみカテゴリには使わない |

アプリ側コンテナ: 今日明日カード48px / ガイド行44px / カテゴリ詳細52px / 品目詳細56px。selfBg系（危険・有害）は透明コンテナでフルサイズ。五十音バッジは28×28px角丸4px。

### 2-4-1. カテゴリ凡例（アプリ・カレンダー下）
色だけに頼らずアイコン+テキストの2チャンネルで伝えるための凡例。カレンダーパネル下部に設置。
- 構成: 小アイコン(20px)+ラベルの横並びチップ。折り返し可、`gap:8px`程度
- **その自治体の`data_{city}.json`に実在するカテゴリのみ**を動的に表示する（全カテゴリ一覧のハードコードは不可。自治体ごとに扱うカテゴリが異なるため）
- 色は各カテゴリのブランドトークン（`TYPE_STYLE`の`fg`/`bg`）を使用し、カテゴリカラーの意味を崩さない
- 影は付けない（タップ不可の表示専用要素のため、2-3節の方針と同じ）

### 2-5. アコーディオン（FAQ）
`<details>/<summary>` を使用（JS不要・a11y確保）。summaryはmin-height 56px、右にexpand_moreアイコン。

### 2-6. クイズ
- 選択肢: `.quiz-choice`（幅100%・min-height 48px・緑枠・左寄せ文字）
- 正解: `.quiz-correct`（ベタ緑+白）／不正解: `.quiz-wrong`（グレー化）
- 解説: `.quiz-answer`（`--gn-bg-note` 淡緑・濃緑文字）
- 回答は `quiz_answer` イベントでGA4送信。1カード1回のみ回答可

### 2-7. 統計カード（ダーク帯）
背景 `rgba(0,0,0,0.18)`・角丸16px。**影は付けない**（タップ不可の表示専用カードのため。v1.4）。数字44px白（Zen Kaku Gothic New）・説明14px `rgba(255,255,255,0.92)` 以上。
**比喩イラスト**: 数字の右に白系ミニSVG（52px前後）を添え、「＝ おにぎり 約4個分」のような換算表記（12px `#A9E7C9`）を付ける。グラフ（棒・円）は使わない — レポート感が出て親しみを損なうため。

### 2-9. 図解ファーストの原則
特徴・使い方などの説明系セクションは「イラスト+短いラベル」を基本形とし、説明文は1行以内に抑える（読む:見る = 4:6）。イラストで伝わるなら説明文は削除する。

### 2-8. 広告枠
プレースホルダは `#F0F2F5` 角丸8px・「広告」ラベル。本番挿入後もコンテンツとの間に24px以上の余白。全画面・アンカー広告は使わない（UX方針）。

### 2-10. 記事ページの見出し階層（h2/h3・v1.13で確定）
記事本文（`public/articles/*.html`）内の見出しは、タグの階層とビジュアルの強弱を分けて設計する。**タグは必ずh1→h2→h3の順で連続させ、飛ばさない**（SEO・スクリーンリーダー対応）。

v1.9でh2/h3ともに下罫線にしたところ「二つとも罫線があると違和感がある」という指摘。v1.10の淡色帯案、v1.11の帯+白文字案を経て、「もう少し工夫したい」と葉っぱアイコン案・吹き出しの尻尾案など複数案を試験導入。参考記事（pa-tu.work/blog/css-fukidashi）のCSS吹き出しパターンも調査し、京葉銀行の単純な三角尻尾とは別パターンを含め計6案を比較した結果、**「帯の左下に、色の濃さが違う丸を大小2つ添える」吹き出し風**で太平さんの最終指定によりv1.13で確定：

```html
<style>
  .h2-decor { position:relative; background:#00885A; color:#fff; border-radius:10px; font-size:18px; padding:12px 16px; font-weight:700; margin-bottom:48px; }
  .h2-decor:before { content:""; position:absolute; top:calc(100% + 3px); left:10px; width:10px; height:10px; border-radius:50%; background:#3eb98f; }
  .h2-decor:after  { content:""; position:absolute; top:calc(100% + 13px); left:22px; width:6px; height:6px; border-radius:50%; background:#88cdb6; }
</style>

<!-- h2: セクション見出し。ブランド色の帯＋左下に丸2つの吹き出し風装飾 -->
<h2 class="sec-title h2-decor">セクションの見出し文</h2>

<!-- h3: セクション内の小見出し。下に点線、上下にゆとりのある余白。本文(16px)より小さくしない -->
<h3 class="text-[16px] font-bold text-[#1C1C1E]" style="border-bottom:2px dotted #6B7280;padding-bottom:8px;margin:32px 0 16px">
  小見出し文
</h3>
```

- h2は「ブランド色の塗りつぶし帯＋白文字＋左下の丸2つ（大10px `#3eb98f`／小6px `#88cdb6`）」で目立たせる。h3は「グレーの点線＋たっぷりの余白（上32px/下16px）」。塗りつぶし面 vs 細い点線というはっきり違う装飾方法にすることで、h2/h3がどちらも「線」だった頃の違和感を解消した
- h3の点線はグレー（`#6B7280`）。h2の帯がブランド色を担うため、h3側は控えめなグレーで階層を譲る
- h2の装飾はどのセクションでも同じ`.h2-decor`クラス1つで済むよう汎用化した。今後の記事②〜⑥でもこのクラスをそのまま使い回す
- 記事内で「なぜ出せないの？」「出し方」「費用の目安」のような定型見出しを繰り返す構成（品目別ガイド系）は、必ずh3タグを使う。太字段落（`<p>`）だけで済ませない
- セクション内を白カード（`bg-white rounded-2xl shadow-sm`）で囲わず、地続きのテキストとして流す（v1.9からの方針を継続）
- セクション同士（`<section>`）の下マージンは`mb-[80px]`
- **「この記事でわかること」「目次」「まとめ」「こんな記事も」のような、記事本文の外側にあるユーティリティ的な要素にも、実体のある見出しとして`<h2>`を使う**（v1.10で追加。h1の直後に見出しなしでいきなり本文相当のブロックが続くのは階層が飛ぶのと同義でSEO上望ましくない）。ビジュアル上は本文セクションのh2（帯スタイル）よりも軽く、15px・帯なしのテキストのみとし、「これは主要な章ではなく付帯情報」という視覚的な弱さを保つ

### 2-11. 記事ページの色運用ルール（プライマリー×グレーのモノトーン・v1.10）
v1.9では「ブランド緑は操作可能要素にのみ使う」というルールにしたところ、実装してみると装飾の大半がインク色（`#1C1C1E`）の塗りつぶし・罫線になり、「全体的に黒々しい」という印象になった。v1.10ではこれを見直し、**インク色は文字色として使うにとどめ、装飾（背景・塗りつぶし・アイコン色）にはブランド緑かグレーのどちらかを使う**という考え方に変更する。

| 役割 | 色 | 用途の例 |
|---|---|---|
| ブランド緑（アクセント） | `#00A86B`（アイコン・ボーダー等）／`#00885A`（h2の帯など濃色塗りつぶし）／淡色`#F3FAF6`（ハイライトボックス背景） | h2の帯背景、ハイライト系ボックス（この記事でわかること／まとめ）の背景・左ボーダー・チェックマーク、リンク・CTAボタン、クリックできるカードのアイコン |
| グレー（ニュートラル） | `#F0F2F5`（面）／`#6B7280`（文字・線） | ナビ・参照系ボックス（目次／こんな記事も）の背景、ステップ番号、目次の番号、h3の点線 |
| インク（文字専用） | `#1C1C1E` | 見出し・本文の文字色。装飾の塗りつぶしや罫線には使わない |

- ユーティリティボックスは「ハイライト系」と「ナビ・参照系」の2種類に分け、色で役割を伝える。**ハイライト系**（この記事でわかること／まとめ）＝記事の要点・結論を凝縮した情報 → 淡いブランド緑の背景＋緑の左ボーダー。**ナビ・参照系**（目次／こんな記事も）＝読者を別の場所に導くだけの情報 → ニュートラルなグレー背景（`#F0F2F5`）
- ページ背景は白のまま維持する（京葉銀行のような清潔感を優先）。そのぶんユーティリティボックスは必ず自身の背景色（上記のグレーまたは淡緑）を持たせ、白いページに埋没しないようにする。**背景が透明・白のままのボックスを作らない**
- 法律・区分の一覧（「根拠法まとめ」のようなカード列）に、ごみの分別カテゴリ色（不燃#4B72D4・資源#00A86B・粗大#6878A0等）を流用しない（v1.9からの方針を継続）
- **h2の帯の背景色は`#00A86B`ではなく`#00885A`を使う**（v1.14で明文化）。`#00A86B`の方が明度は高く明るく見えるが、白文字を乗せたときのコントラスト比は`#00A86B`が3.08:1、`#00885A`が4.50:1で、本書が定める4.5:1基準（3章参照）を満たすのは`#00885A`のみ。「明るさ」より「白文字での可読性」を優先する

### 2-12. ヘッダー／フッターは背景を画面幅いっぱいに、コンテンツは中央寄せ（v1.11）
`<header>`と`<footer>`は、背景色・境界線（border-top/border-bottom）を画面幅いっぱいに表示し、その内側にコンテンツ幅を制限した`<div>`を置く二重構造にする。**`max-w-*` や `mx-auto` は内側の`<div>`にだけ付け、`<header>`/`<footer>`要素自体には付けない**（v1.11で修正。以前はfooterに`max-w-xl mx-auto`を直接付けていたため、デスクトップ幅で見るとheaderの帯は画面いっぱいなのにfooterの帯だけ中央の細い帯になり、左右非対称に見えるバグがあった）。

```html
<header class="bg-white border-b border-black/[0.07]">
  <div class="max-w-xl mx-auto px-5 py-4">…</div>
</header>

<footer class="bg-white border-t border-black/[0.05]">
  <div class="max-w-xl mx-auto text-center px-6 py-8">…</div>
</footer>
```

### 2-13. 本文中の語句ハイライト（v1.14で判断基準を厳格化）
記事本文（`.body-p`）は文章量が多く、流し読みする読者が要点を拾いにくい。**段落中で本当に重要な語句（法律の正式名称・費用・重要な義務や警告）に限り、`<mark>`タグでマーカー風のハイライトを敷く**。

```html
<style>
  mark { background:#EAF6F0; color:#1C1C1E; padding:1px 4px; border-radius:4px; font-weight:700; }
</style>

<p class="body-p"><mark>家電リサイクル法</mark>により、テレビ・冷蔵庫…</p>
```

- 使うのは`<mark>`（意味のある強調というセマンティクスを持つタグ。装飾目的の`<span>`より適切）。色はブランド緑の淡色`#EAF6F0`＋インク文字色（2-11のハイライト系ボックスと同じ色相で統一感を持たせる）
- **1段落につき1箇所まで**。語句も短く（3〜10文字程度の名詞・キーフレーズ）。文単位・複数箇所への多用は「モノトーンで整理する」という2-11の方針と矛盾するため禁止
- ハイライト対象は次の3種類**のみ**。①法律・制度の正式名称の初出（例:「家電リサイクル法」）②費用・料金（例:「数百円程度」）③「絶対にやってはいけない」「義務がある」等の、それを守らないと実害が出る警告・義務表現（例:「絶対に出せません」「引取義務」）
- **NG例（v1.14で撤去）**: 「有償」のように文脈なしで単語だけ浮いているもの／「データを消去」「複数社へ相談」のような“やっておくと安心な一般的なアドバイス”。これらは③の警告・義務ほどの重要度がなく、単なる強調に流れやすいため対象外とする。**該当する語句が無理に見つからない段落は、無理にハイライトを作らずゼロ件のままでよい**（1記事内で偏りがあっても問題ない）

## 3. アクセシビリティ基準（デジタル庁デザインシステムの考え方を参照）

1. **コントラスト**: テキスト4.5:1以上（18px超の太字は3:1）、UIパーツ3:1以上。ダーク帯上の薄色文字は白ベース（rgba白≥0.9）を使う
2. **タッチターゲット**: 最小44×44px、主要操作は48px推奨。視覚サイズが小さくても余白で領域を確保
3. **色だけに頼らない**: 状態・カテゴリは必ず「色+アイコン or テキスト」の2チャンネルで伝える
4. **フォーカス可視**: `outline: none` 単独使用の禁止。キーボード操作でフォーカスリングが見えること
5. **セマンティクス**: header/main/section/button等を正しく使う。装飾画像は `alt=""`、意味のあるSVGは `role="img"`+`aria-label`
6. **動きへの配慮**: 自動再生・過剰アニメーション禁止。動きを付ける場合は `prefers-reduced-motion` でフォールバック
7. **文字の拡大**: ユーザーのズーム・文字拡大を妨げる指定をしない

## 5. 運用ルール

1. **変更はDS→実装の順**: 見た目のルールを変えるときは、先に本書を更新してから実装する（差分が仕様変更なのか逸脱なのかを明確にするため）
2. **検品チェックリスト（厳格さん）**: ①タグバランス ②コントラスト ③タッチターゲット ④8px倍数の余白 ⑤カテゴリ色の誤用なし ⑥トークン外の色・サイズが増えていないか
3. **script.js／キャッシュ**: 見た目に関わるJS・CSS変更時は sw.js の CACHE_NAME を上げる
4. **統計数値の鮮度**: 数字セクションの出典データは年1回（毎年3〜6月頃の公表後）に更新チェック
5. **新自治体追加時**: 色は `data_{city}.json` の `brand_color` のみで差別化。構造・コンポーネント・カテゴリアイコン（`TYPE_STYLE`の`img`）は自治体ごとに変更しない（経緯は1-2節参照）
6. **`garbage_db`のAI推定品目（v1.7・2026-07-21）**: 自治体の公式資料に明記のない品目を追加する場合は、全国的にほぼ一律の扱いとなる日用品に限定し、`"source": "estimated"`を必ず付与する。危険ごみ・有害ごみ・ごみ出し不可・粗大ごみ等、誤ると実害（発火・けが・手数料トラブル等）につながるカテゴリでは推定追加を行わない。`openItemDetail()`が`source: "estimated"`を検出すると、公式資料に無い旨と環境推進課の問い合わせ先を自動表示する仕組みが既に実装済み
